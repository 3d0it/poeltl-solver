# Poeltl Solver

[![CI](https://github.com/3d0it/poeltl-solver/actions/workflows/ci.yml/badge.svg?branch=main)](https://github.com/3d0it/poeltl-solver/actions/workflows/ci.yml)

A .NET console solver for a Poeltl-like NBA player guessing game.

The app reads the included `data/poeltl_players.csv` snapshot, accepts the feedback you get from Poeltl after each guess, keeps a full feedback history, and ranks future guesses using an entropy-based scoring algorithm.

## Project Layout

```text
PoeltlSolver/
  ConsoleApp/          Console input/output and player-name completion

PoeltlSolver.Core/
  Data/                CSV loading
  Feedback/            Poeltl feedback model, parsing, and comparison rules
  Models/              Player model
  Solver/              Candidate filtering and entropy scoring

PoeltlSolver.Tests/    xUnit test project
data/
  download_players.py  NBA.com roster downloader
  poeltl_players.csv   Versioned player snapshot used by the solver
```

## Quick Start

The repository includes a `data/poeltl_players.csv` snapshot, so the solver can run immediately after clone.

```bash
dotnet build PoeltlSolver.sln
dotnet run --project PoeltlSolver
dotnet test PoeltlSolver.sln
```

The same build and test commands are also run by the GitHub Actions CI workflow.

## Refreshing the Dataset Snapshot

Refreshing the snapshot is optional. The committed CSV is enough to build, test, and run the app; refresh it when you want newer NBA roster data.

The dataset snapshot is generated by:

```bash
python3 data/download_players.py --season 2025-26
```

The script downloads NBA rosters from NBA.com through `nba_api`, enriches each player with conference and division metadata, and writes:

```text
data/poeltl_players.csv
```

Expected CSV columns:

```text
Name,Team,Conference,Division,Position,Height,HeightInches,Age,Number
```

Important limitation: the dataset only contains each player's current team. Poeltl's yellow team feedback means "the target previously played for that team", but this CSV does not include historical teams.

The committed snapshot can differ from Poeltl's live data and NBA.com after roster, jersey, height, or age changes. Regenerating the CSV updates the snapshot, but it is intentionally not part of CI so tests do not depend on NBA.com availability.

This project is not affiliated with Poeltl, the NBA, or NBA.com.

## Running the Console App

From the repository root:

```bash
dotnet run --project PoeltlSolver
```

The console shows:

- current candidate count
- top information guesses
- top candidate guesses, once the candidate set has been reduced
- a prompt for the player you guessed
- a compact feedback prompt

Player-name input supports `Tab` completion.

Example:

```text
Guess player name: Zio<Tab>
```

can complete to:

```text
Zion Williamson
```

## Feedback Input Format

Enter feedback as one 7-character string:

```text
Team Conf Div Pos Ht Age #
```

Symbols:

```text
= correct / equal
- wrong
~ close
> higher
< lower
+ close higher
_ close lower
```

For example:

```text
-=-->_>
```

means:

```text
Team wrong
Conference correct
Division wrong
Position wrong
Height higher
Age close lower
Number higher
```

For numeric fields, direction is relative to the target:

- `>` means the target value is higher than your guess
- `<` means the target value is lower than your guess
- `+` means the target value is higher and within 2
- `_` means the target value is lower and within 2

## Feedback Rules

The solver models these Poeltl rules:

- Green is an exact match.
- Position yellow means the guess and target share at least one position.
- Height, age, and number yellow means the value is off by at most 2.
- Numeric arrows indicate whether the target is higher or lower than the guess.

Position matching is set-based:

```text
G-F vs F-G -> Green
G   vs G-F -> Yellow
F   vs G-F -> Yellow
G   vs F   -> Gray
```

## Solver Algorithm

The solver is a hybrid exact/soft entropy-based solver.

At each turn, it evaluates possible guesses by simulating that guess against every current candidate target:

```text
FeedbackEngine.Compare(guess, candidate)
```

Candidates that would produce the same feedback are grouped together. Each group represents one possible result you could receive from Poeltl.

Before exact filtering or scoring, feedback is normalized into the solver's local data model:

- `Team -` and `Team ~` both mean "not the guessed current team" because previous-team history is unavailable.
- Non-green height feedback is grouped by arrow direction when a direction exists.
- Position is not used as an eliminating constraint because live game position labels can differ from the local CSV.
- Conference, division, age, and number remain strict for exact candidates.

The solver keeps the full guess history and recomputes compatibility from scratch after each feedback entry. If exact candidates exist, those are shown as possible solutions. If exact candidates collapse to zero, the solver switches to near-match mode instead of pretending the game is unsolvable.

Near-match mode assigns weighted penalties:

- Low penalty: position mismatch, small height drift, or height color drift with the same arrow direction.
- Medium penalty: team-history ambiguity.
- High penalty: conference, division, age, or number mismatch.

Near matches include the mismatched fields so you can see whether the issue looks like dataset drift or an input mistake.

Entropy is computed as:

```text
H = -sum(p * log2(p))
```

where `p` is:

```text
group size / total candidates
```

A high entropy score means the guess tends to split the candidate space into many informative groups.

The solver also computes expected remaining candidates:

```text
sum(p * group size)
```

The ranking order is:

1. higher entropy
2. lower expected remaining candidates
3. guesses that are still possible candidates
4. player name

The app prints two lists:

- `Top information guesses`: can include non-candidates if they are useful probes
- `Top candidate guesses`: only players still possible as the solution

## Heuristics and Data Mismatch Handling

Poeltl live data can differ slightly from the local CSV, especially for height and position. To avoid eliminating valid targets too aggressively, the solver applies targeted heuristics:

- For exact candidates, height green remains strict.
- For exact candidates, non-green height feedback is grouped by arrow direction when a direction exists.
- If exact candidates collapse, a one-inch height mismatch against a green height is treated as a low-penalty near match.
- Position mismatches are treated as low-penalty near-match evidence, not as hard eliminators.

Example:

```text
CSV says 6-5
Poeltl says 6-4
```

That can change a height result from gray to yellow, or yellow to gray, while preserving the arrow. The solver keeps such candidates.

Age and number remain strict for exact candidates because they are less ambiguous in the current dataset. In near-match mode they are still allowed, but carry a high penalty.

If exact candidates collapse to zero, the solver shows the best near matches with their mismatch reasons instead of restoring incompatible candidates.

Jersey numbers preserve their displayed label because `0` and `00` are different NBA jersey numbers. Exact number feedback requires the label to match, while arrows and close feedback use the numeric value.

## Design Tradeoffs

- Height matching is fuzzy for gray/yellow when the arrow direction matches because Poeltl's live height source can differ from NBA.com by an inch.
- `Team -` and `Team ~` are equivalent in the local solver because previous-team history is not in the CSV. Both are treated as "not the guessed current team"; position is non-eliminating, while conference, division, height, age, and number still constrain exact candidates.

## Known Limitations

- The local dataset can differ from Poeltl's live source. Height is the most visible mismatch, so the solver has a targeted height-direction heuristic.
- Previous teams are not included in `data/poeltl_players.csv`, so team feedback is current-team based rather than historical.
- The solver is greedy: it optimizes the next guess by entropy and does not search the full multi-turn decision tree.
- The included CSV is a versioned snapshot; refresh it manually when you want newer roster data.

## Tests

The repository includes an xUnit test project for the solver core.

Run tests with:

```bash
dotnet test PoeltlSolver.sln
```

The tests cover:

- position matching
- numeric close matching
- compact feedback parsing
- CSV loading
- jersey number parsing, including `00`
- height color-drift tolerance
- team feedback normalization
- entropy scoring over normalized feedback groups
- zero-candidate near-match behavior
